### `convert.js` – key design decisions

**Why a `scope`?**  
ArkType's `match({cases})` parses every *string key* as an ArkType type expression. String literal union-object types (`{ kind: "length", meters: number }`) cannot be embedded as plain JS string keys. The clean solution is to define a `scope` whose **exported names become valid string keys** for `match` — keeping dispatch fully type-driven while staying within ArkType's API.

**The `match({...})({...})` two-call pattern:**  
```js
const convert = conversions.match({      // ← first ({...})
  length:      (d) => `${(d.meters     * 3.28084).toFixed(2)} ft`,
  mass:        (d) => `${(d.kilograms  * 2.20462).toFixed(2)} lb`,
  temperature: (d) => `${((d.celsius * 9) / 5 + 32).toFixed(0)} °F`,
})({ default: "assert" });               // ← second ({...})
```
The first call accumulates the three branches and returns the chained parser (no `default` → doesn't finalize). The second call seals the union with `default: "assert"`, causing ArkType to throw a `TraversalError` for any unmatched input.
