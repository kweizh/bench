### Key design decisions

**Why `scope` + named types as match keys?**  
ArkType's `match({})` parses each non-`"default"` key through its type parser. The standard scope doesn't support inline object-literal syntax (e.g. `'{ status: "success" }'`) as a string key. Using `scope({ success: ..., error: ..., pending: ... })` registers named types so they resolve as plain identifiers (`success`, `error`, `pending`) inside `match({})`. The resulting call pattern is:

```ts
const handlePayload = payloadScope.type.match({   // match({...})
  success:  (v) => `OK: ${JSON.stringify(v.data)}`,
  error:    (v) => `ERR ${v.code} ${v.reason}`,
  pending:  ()  => "PENDING",
  default:  "assert",                              // throws on no match
});

handlePayload(input);                              // ({...}) – second call
```

This is the `match({...})({...})` call signature required by criterion 5.
