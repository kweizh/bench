# Custom OTel Tracing of an LLM Agent with the Langfuse JS/TS SDK ## Background You are building a small TypeScript Node.js script that simulates a weather assistant. The team wants every run of this script to produce a fully structured Langfuse trace so they can inspect the model call inside the Langfuse UI. Langfuse's JS/TS SDK is OpenTelemetry-native: spans created with `@langfuse/tracing` are forwarded to Langfuse by the `LangfuseSpanProcessor` from `@langfuse/otel`. This task evaluates whether you can correctly set up OpenTelemetry, create a parent observation and a nested generation observation, and properly flush the spans before the process exits so that they actually arrive in the Langfuse backend. ## Requirements * Build a Node.js + TypeScript project under `/home/user/myproject`. * Initialize the OpenTelemetry Node SDK with the Langfuse `LangfuseSpanProcessor` so that spans are exported to Langfuse. * Create a single root observation ("span") that represents the user request. * Inside the root observation, create a nested observation of type `generation` that represents the simulated LLM call (no real model call is needed, simulate the output in code). * Tag both observations with the appropriate input/output payloads and the generation must include a model name attribute. * Ensure spans are reliably flushed to Langfuse before the process exits (the script is short-lived). * Print the resulting trace ID to standard output and to a log file so a human reviewer can find the trace in Langfuse. ## Implementation Hints * The required packages are `@langfuse/tracing`, `@langfuse/otel`, and `@opentelemetry/sdk-node`. Pin or install the latest 4.x line of `@langfuse/*` packages. * Use `startActiveObservation` from `@langfuse/tracing` to open the root span and `startObservation` (with `{ asType: "generation" }`) to open the nested generation. * The OTel SDK must be started before any module that emits spans is imported. The `LangfuseSpanProcessor` reads `LANGFUSE_PUBLIC_KEY`, `LANGFUSE_SECRET_KEY`, and `LANGFUSE_BASEURL`/`LANGFUSE_BASE_URL` from the environment. * Short-lived processes drop spans unless they call `await sdk.shutdown()` (or an equivalent flush) before exiting. Make sure this happens after the trace is fully ended. * You can obtain the current trace ID via `getActiveTraceId` from `@langfuse/tracing` while inside `startActiveObservation`. * Read `run-id` from the `ZEALT_RUN_ID` environment variable and include it in the parent observation name so concurrent task runs do not collide. ## Acceptance Criteria * Project path: `/home/user/myproject` * Ensure the real script is executed at least once and its artifacts exist before verification runs. * The script must be runnable from the project root with `npm run start`. * Log file: `/home/user/myproject/output.log` * The script must read `run-id` from the `ZEALT_RUN_ID` environment variable. All Langfuse-side names that are checked by the verifier must use this `run-id`. * After the script finishes, the log file must contain the resulting trace ID on a dedicated line in the format: `Trace ID: <trace_id>`. * A trace whose `name` is exactly `weather-assistant-${run-id}` must exist in the configured Langfuse project, reachable via the Langfuse Public API at `GET /api/public/traces` (filtered by `name`) and at `GET /api/public/traces/{traceId}`. * The trace must contain at least one nested observation of type `GENERATION` named `llm-call`, with the `model` field set to a non-empty string. * Both the root observation and the nested generation must have non-null `input` and `output` payloads as returned by `GET /api/public/traces/{traceId}` (i.e. the `observations` array in the response).
