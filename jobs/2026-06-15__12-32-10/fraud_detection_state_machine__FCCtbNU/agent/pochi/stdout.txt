## State Machine Logic

Each user is tracked independently through three states: `LOGGED_OUT` → `LOGGED_IN` → `SUSPICIOUS`:

| Event | Condition | Transition |
|-------|-----------|------------|
| `login` | — | → `LOGGED_IN` (record timestamp, reset counter) |
| `transaction` | amount ≥ 1000, within 300s of login, count < 2 | → `SUSPICIOUS` (increment counter) |
| `transaction` | amount ≥ 1000, within 300s of login, count = 2 | → `LOGGED_OUT` (emit `FRAUD_ALERT`, reset) |
| `transaction` | > 300s since login | → `LOGGED_OUT` (ignore transaction) |
| `transaction` | amount < 1000 | No state change |
| `transaction` | while `LOGGED_OUT` | No state change |
| `logout` | — | → `LOGGED_OUT` (reset) |

## Key Implementation Details

- Uses `op.key_on()` to partition events by `user_id`
- Uses `op.stateful_map()` to maintain per-user state immutably (returns new state dict each time)
- State is `None` initially, lazily initialized to `LOGGED_OUT`
- Fraud alerts are emitted as `(user_id, alert_dict)` tuples, filtered for non-None, and written as JSONlines
