# Track LLM Generation Token Usage and Cost with the Langfuse Python SDK ## Background When building production LLM applications, accurately tracking per-call token usage and cost is essential for billing, capacity planning, and cost optimization. https://langfuse.com/ supports first-class **token & cost tracking** on observations of type `generation` (and `embedding`) via the `usage_details` and `cost_details` parameters, which can be **ingested directly** by your code instead of being inferred from a model definition. In this task, you will write a small Python script that simulates an LLM call and creates a Langfuse `generation` observation that captures the exact token usage and per-token USD cost reported by the (simulated) provider. The generation must be nested inside a span so that the resulting trace has the structure parent-span → child-generation. After the data has been ingested, your script must flush the SDK and persist the resulting `trace_id` to a log file so that an external verifier can fetch the trace from the Langfuse Public API and confirm that the usage and cost fields were ingested correctly. ## Requirements * Implement a single-shot Python script that uses the Langfuse Python SDK (v3+, package `langfuse`) and reads credentials from the `LANGFUSE_PUBLIC_KEY`, `LANGFUSE_SECRET_KEY`, and `LANGFUSE_BASE_URL` environment variables. * Read `run-id` from the `ZEALT_RUN_ID` environment variable and include it in the trace name so that concurrent runs do not collide. * Create one parent observation of type `span` named `chat-pipeline-${run-id}` that becomes the root of the trace. * Inside the span, create one nested observation of type `generation` named `chat-completion`. The generation MUST set: * `model` = `"gpt-4o-mini"` * `input` = a list of chat messages of the form `[{"role": "user", "content": "Summarize Langfuse in one sentence."}]` * `output` = the string `"Langfuse is an open-source LLM engineering platform for observability, prompts, and evals."` * `usage_details` containing the integer keys `input`, `output`, and `total` with the exact values `25`, `40`, and `65` respectively * `cost_details` containing the numeric keys `input`, `output`, and `total` with the exact USD values `0.000125`, `0.00040`, and `0.000525` respectively * Call `langfuse.flush()` (or otherwise block until ingestion has been queued) before the script exits so the data is delivered to the Langfuse backend. * Write the resulting Langfuse trace ID to the log file in the format `Trace ID: <trace_id>` (single line, no extra characters around the ID). ## Implementation Hints * The Python SDK exposes a singleton client through `from langfuse import get_client`; once env vars are set the client picks them up automatically. * The primary API for nested instrumentation is the `start_as_current_observation(as_type=..., name=...)` context manager. Use `as_type="span"` for the parent and `as_type="generation"` for the child. * `usage_details` and `cost_details` are passed as plain `dict`s and can be supplied either when starting the generation or later via `generation.update(...)`. Langfuse will only persist them if the observation type is `generation` (or `embedding`). * Use `langfuse.get_current_trace_id()` (called from inside the active span) to obtain the trace ID that the verifier will look up; do **not** invent an ID yourself. * Read `run-id` from `os.environ["ZEALT_RUN_ID"]` and apply it consistently to the span name. ## Acceptance Criteria * Project path: /home/user/myproject * Ensure the script is executed and the artifacts exist (a real trace is ingested into Langfuse Cloud and the log file is written). * Log file: /home/user/myproject/output.log * The log file must contain exactly one line in the format `Trace ID: <trace_id>` where `<trace_id>` is the Langfuse-assigned trace identifier returned by the SDK. * The ingested trace must contain an observation of type `GENERATION` named `chat-completion` whose `providedModelName` is `gpt-4o-mini`, whose `usageDetails` map exposes the exact values for the keys `input`, `output`, and `total`, and whose `costDetails` map exposes the exact USD values for the same three keys (as documented in the Requirements section). * The ingested trace must contain a parent observation of type `SPAN` whose name is `chat-pipeline-${run-id}` where `run-id` is the value read from the `ZEALT_RUN_ID` environment variable.
