# Redact PII Before Sending Traces to Langfuse (Python SDK) ## Background Your team is shipping an LLM application that processes customer support messages containing personally identifiable information (PII) such as email addresses, phone numbers, and credit-card numbers. Compliance requires that this data never reach the Langfuse server, yet you still want full trace visibility for debugging. Langfuse supports a server-side masking pattern: the Python SDK accepts a `mask` callable that is applied to every observation's `input`, `output`, and `metadata` **before** the event is exported. Your job is to implement that masking and prove via the Langfuse public API that the sensitive data is gone from the trace. ## Requirements - Create a Python project at `/home/user/myproject` with a runnable script `run.py`. - The script must initialize the Langfuse Python SDK with a custom `mask` function that, when applied to string data, replaces: - email addresses with the literal token `[REDACTED EMAIL]` - North American phone numbers in the form `NNN-NNN-NNNN` (with `-`, `.`, or space separators) with the literal token `[REDACTED PHONE]` - 13–19 digit credit-card numbers (groups optionally separated by spaces or dashes) with the literal token `[REDACTED CREDIT CARD]` - The script must create exactly one Langfuse trace whose top-level (root) observation: - has the name `mask-demo-${run-id}` where `run-id` is read from the `ZEALT_RUN_ID` environment variable, - has an `input` and `output` that, **before masking**, contain at least one email address, one phone number, and one credit-card number (all three PII types must appear together in both `input` and `output`). - The script must create at least one **child generation observation** under the root span; its `input` and `output` must also contain at least one of the PII types above before masking. - After all observations are recorded, the script must flush events to Langfuse so the data is persisted server-side. - The script must write the trace identifier to a log file using the exact format described in Acceptance Criteria. ## Implementation Hints - Install the Langfuse Python SDK (`langfuse`) into the project's virtual environment. - Langfuse credentials are already exported as `LANGFUSE_PUBLIC_KEY`, `LANGFUSE_SECRET_KEY`, and `LANGFUSE_BASE_URL`. - Configure masking by passing a `mask=` callable when constructing the `Langfuse` client, then obtain the active client (e.g. via `get_client()`) and use its context-manager APIs to create the span and nested generation. - The mask callable receives the raw value (which may be a string, dict, or list) and should return a sanitized value of the same shape. Recurse into dicts and lists so nested fields are also redacted. - The trace ID is the OpenTelemetry trace ID of the root observation. You can obtain it from the span object returned by Langfuse's context manager (e.g. via `trace_id` on the span wrapper). - Remember to flush before the process exits, otherwise the events are dropped silently. ## Acceptance Criteria - Project path: `/home/user/myproject` - Ensure the script is executed and the artifacts exist. - Log file: `/home/user/myproject/output.log` - The script entry point must be runnable as `python3 /home/user/myproject/run.py`. - The root trace observation name must be `mask-demo-${run-id}` where `run-id` is read from the `ZEALT_RUN_ID` environment variable. - After the script runs, the log file must contain exactly one line in the format: `Trace ID: <trace_id>` where `<trace_id>` is the Langfuse / OpenTelemetry trace ID of the recorded trace. - When fetching the trace via the Langfuse public API endpoint `GET /api/public/traces/{traceId}`, the returned trace and its observations must satisfy all of the following: - The trace `name` equals `mask-demo-${run-id}`. - The trace `input` and `output`, when serialized to a string, contain the literal tokens `[REDACTED EMAIL]`, `[REDACTED PHONE]`, and `[REDACTED CREDIT CARD]`. - The trace `input` and `output`, when serialized to a string, contain none of the original PII values that the agent's script fed in (i.e. no raw email, phone, or credit-card digits leak through). - At least one observation of type `GENERATION` exists, and its `input` and `output`, when serialized to a string, also contain at least one of the `[REDACTED EMAIL]`, `[REDACTED PHONE]`, or `[REDACTED CREDIT CARD]` tokens and none of the original PII values.
