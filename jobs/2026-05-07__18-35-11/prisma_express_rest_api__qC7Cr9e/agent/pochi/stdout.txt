## What was created

### `prisma/schema.prisma`
Defines two models:
- **User** â€” `id`, `email` (unique), `name`, and `tasks` relation
- **Task** â€” `id`, `title`, `description?`, `status` (default `"todo"`), `priority` (default `0`), `userId` FK, `createdAt`, `updatedAt`

### `server.js`
Full Express v5 CRUD API with input validation and proper HTTP status codes:

| Method | Endpoint      | Description |
|--------|---------------|-------------|
| POST   | `/users`      | Create user; 409 on duplicate email |
| POST   | `/tasks`      | Create task with user embedded in response |
| GET    | `/tasks`      | List all tasks (ordered by priority desc); supports `?status=` and `?userId=` filters |
| GET    | `/tasks/:id`  | Single task with user included |
| PATCH  | `/tasks/:id`  | Partial update â€” any subset of `title/description/status/priority` |
| DELETE | `/tasks/:id`  | Delete; returns 204 No Content |

## Key implementation notes
- **Prisma v7** changed the architecture: the `url` is no longer accepted in `schema.prisma`. Connection is established via the `@prisma/adapter-better-sqlite3` driver adapter passed to `new PrismaClient({ adapter })`.
- Migration was run with `npx prisma migrate dev --name init`, creating the SQLite database at `./dev.db`.
- Graceful shutdown disconnects the Prisma client on SIGTERM/SIGINT.
- Artifacts saved to `/logs/artifacts/`.
