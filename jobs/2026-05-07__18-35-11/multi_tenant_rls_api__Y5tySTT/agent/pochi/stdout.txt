### What was built

**`prisma/schema.prisma`** — `Item` model with `id`, `name`, `tenantId` fields and an index on `tenantId`.

**`server.js`** — Express API with tenant isolation enforced via `prisma.$extends`:

| Route | Description |
|-------|-------------|
| `GET /items` | Returns only items where `tenantId` matches `x-tenant-id` header |
| `POST /items` | Creates an item; `tenantId` is automatically set from `x-tenant-id` header |

---

### Tenant Isolation Mechanism

A `createScopedClient(tenantId)` function is called per-request. It uses `basePrisma.$extends({ query: { item: { ... } } })` to intercept every `item.*` operation and inject the caller's `tenantId` into both `where` clauses (reads) and `data` payloads (writes). Cross-tenant data access is impossible at the ORM level regardless of what query arguments are passed in.

```
Middleware → x-tenant-id header → createScopedClient(tenantId) → req.prisma
                                         ↓
                               $extends intercepts all item queries
                               injects tenantId into where / data
```

---

### Verified behaviour (curl tests)

```bash
# POST creates items for respective tenants
POST /items  x-tenant-id: tenant-a  {"name":"Widget Alpha"}  → 201 + tenantId:"tenant-a"
POST /items  x-tenant-id: tenant-b  {"name":"Widget Gamma"}  → 201 + tenantId:"tenant-b"

# GET returns only matching tenant's items
GET /items   x-tenant-id: tenant-a  → ["Widget Alpha", "Widget Beta"]  (tenant-b item absent)
GET /items   x-tenant-id: tenant-b  → ["Widget Gamma"]                 (tenant-a items absent)

# Error handling
GET /items   (no header)            → 400 "Missing or empty x-tenant-id header"
POST /items  (no name in body)      → 400 "Request body must include a non-empty name field"
```
