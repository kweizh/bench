### Architecture

```
Request
  └─► pbAuthMiddleware
        ├─ new PocketBase(PB_URL)          — fresh instance per request (no shared state)
        ├─ pb.authStore.loadFromCookie()   — hydrates auth store from incoming cookie header
        ├─ pb.collection('users').authRefresh()  — validates & refreshes token via PocketBase API
        │     └─ on error → pb.authStore.clear()  — wipes store so exported cookie is empty
        ├─ res.setHeader('Set-Cookie', pb.authStore.exportToCookie())  — always writes back
        └─ res.locals.pb = pb             — passes instance to route handlers
  └─► GET /profile
        ├─ pb.authStore.isValid + record present → 200 { id, email }
        └─ otherwise → 401 { error: "Unauthorized" }
```

### Key design decisions

| Decision | Reason |
|---|---|
| `require('pocketbase/cjs')` | Project uses `"type": "commonjs"`, so the ESM-first default export can't be used with `require()`. The `/cjs` subpath export resolves to `pocketbase.cjs.js`. |
| Fresh `PocketBase` instance per request | Prevents auth state leaking between concurrent requests (the SDK stores auth in-memory on the instance). |
| `authRefresh()` only when `authStore.isValid` | Avoids a redundant network round-trip for anonymous requests. |
| `pb.authStore.clear()` on refresh failure | Ensures `exportToCookie()` produces an expired/empty cookie so the client is correctly de-authenticated. |
| `Set-Cookie` written unconditionally | Both the 200 and 401 paths always carry a `Set-Cookie` header, satisfying the requirement in both branches. |

### Verified behaviour

| Scenario | Status | Set-Cookie |
|---|---|---|
| No cookie | 401 `{"error":"Unauthorized"}` | Cleared cookie (`Expires: epoch`) |
| Invalid / expired token | 401 `{"error":"Unauthorized"}` | Cleared cookie (`Expires: epoch`) |
| Valid `pb_auth` cookie | 200 `{"id":"…","email":"…"}` | Refreshed cookie with new token & future expiry |
