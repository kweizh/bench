# Sequelize Hooks and Internal Transactions ## Background Sequelize may use transactions internally for operations like `Model.findOrCreate`. If your hook functions execute read or write operations that need to be atomic with the main operation, you must explicitly pass the transaction object to those operations. ## Requirements * Create a Node.js module in `index.js` that initializes a Sequelize SQLite database with two models: `User` and `AuditLog`. * `User` model: `username` (STRING, unique), `status` (STRING). * `AuditLog` model: `action` (STRING), `username` (STRING). * Define a `beforeCreate` hook on the `User` model that creates an `AuditLog` entry with `action: 'Creating user'` and the user's `username`. * Define an `afterCreate` hook on the `User` model that throws a specific error (`new Error('Simulated failure')`) if the `username` is exactly `'error_user'`. * The `beforeCreate` hook **must** participate in the same transaction as the `User` creation. If the `User` creation fails (e.g., due to the `afterCreate` hook throwing an error), the `AuditLog` creation must also be rolled back. * Export an async function `initDB()` that syncs the database (e.g., `sequelize.sync({ force: true })`). * Export an async function `runFindOrCreate(username)` that calls `User.findOrCreate({ where: { username }, defaults: { status: 'active' } })`. It should catch and return any errors thrown during the process, or return the created user. ## Implementation Hints * When `findOrCreate` is executed, Sequelize creates an internal transaction. * In your `beforeCreate` hook, check for `options.transaction` and pass it to the `AuditLog.create` method. * If you don't pass the transaction, the `AuditLog` entry will be committed immediately, bypassing the rollback if the main transaction fails. ## Acceptance Criteria * Project path: /home/user/myproject * Command: `node index.js` (can be used to test locally, though verification will require the module directly) * Exports: The module must export `initDB`, `runFindOrCreate`, `User`, and `AuditLog`. * Successful creation: Calling `runFindOrCreate('normal_user')` creates both a `User` and an `AuditLog`. * Transaction rollback: Calling `runFindOrCreate('error_user')` throws/returns an error. The `AuditLog` for `'error_user'` must **not** exist in the database because the transaction was rolled back.
