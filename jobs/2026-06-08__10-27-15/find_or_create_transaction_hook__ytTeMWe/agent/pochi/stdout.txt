This means the `AuditLog` row written by the `beforeCreate` hook is permanently persisted — exactly the wrong behaviour. The fix is to supply an **explicit** transaction to `findOrCreate`. ### Root cause & fix

Sequelize v6's internal `findOrCreate` transaction has a bug: its `finally` block always **commits**, even when an error propagates out of a hook. Supplying an explicit transaction (`transaction: t`) sets `internalTransaction = false`, disabling that auto-commit so we control commit/rollback ourselves.

```js
async function runFindOrCreate(username) {
  const t = await sequelize.transaction();
  try {
    const [user] = await User.findOrCreate({
      where: { username },
      defaults: { status: 'active' },
      transaction: t,          // ← we own this transaction
    });
    await t.commit();
    return user;
  } catch (err) {
    await t.rollback();        // ← rolls back AuditLog too
    return err;
  }
}
```

The `beforeCreate` hook forwards `options.transaction` to `AuditLog.create`, making both writes atomic:

```js
User.addHook('beforeCreate', async (user, options) => {
  await AuditLog.create(
    { action: 'Creating user', username: user.username },
    { transaction: options.transaction }   // ← same transaction
  );
});
```
