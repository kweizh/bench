# PocketBase Atomic Wallet Withdrawal Hook (Go) ## Background https://pocketbase.io/ is a single-file Go backend that uses an embedded SQLite database in WAL mode. SQLite WAL allows many concurrent readers but only a single writer. Because of this, any database mutation that has to span more than one row (such as moving money from a wallet to a ledger entry) must run inside a single transaction, and inside the transaction callback **every** database operation has to go through the transaction-scoped `core.App` instance that PocketBase hands to the callback (commonly named `txApp`). Reusing the global `app` instance from inside the transaction will deadlock the writer. Your job is to extend a pre-built PocketBase v0.31.0 Go application that already provisions a `wallets` collection and a `withdrawals` collection via a registered migration. You will add a **Go event hook** that wraps record creation on the `withdrawals` collection so that a withdrawal is only accepted when the wallet has sufficient funds, and so that the wallet's balance is debited **atomically** with the persisted withdrawal row. ## Requirements * Register a Go event hook on the public **record create request** for the `withdrawals` collection (i.e. wrap the REST `POST /api/collections/withdrawals/records` endpoint, not internal `app.Save` calls). * Validate the incoming payload: * The `amount` field must be present and strictly greater than `0`. Otherwise reject the request with HTTP `400 Bad Request` and DO NOT persist anything. * The `wallet` field must reference an existing `wallets` record. Otherwise reject with HTTP `400 Bad Request` and DO NOT persist anything. * The wallet's current `balance` must be greater than or equal to the submitted `amount`. Otherwise reject with HTTP `400 Bad Request` and DO NOT persist anything and DO NOT modify the wallet. * On valid input, run a single PocketBase transaction in which: * The referenced wallet's `balance` is decreased by exactly `amount`. * The withdrawal record is persisted with the submitted `wallet`, `amount`, and optional `note` values. * If any step inside the transaction fails, both the balance change AND the withdrawal record must be rolled back so the database stays consistent. * The hook must continue to propagate the request chain so that PocketBase's default save logic runs for the withdrawal record; forgetting to propagate is a known v0.23+ pitfall and will block all writes. * Keep the resulting binary runnable as a standalone PocketBase backend (`./wallet-app serve --http=0.0.0.0:8090`). Do not remove the existing migration registration or change the schema of either collection. ## Implementation Hints * PocketBase v0.31 uses the chained hook API. Register your handler with `app.OnRecordCreateRequest("withdrawals").BindFunc(...)` and remember the post-v0.23 propagation rule: return `e.Next()` from the success path or the chain is silently aborted. * The transaction-scoped app instance is exposed via `app.RunInTransaction(func(txApp core.App) error { ... })`. Inside the callback you **must** use `txApp` (not the captured outer `app` / `e.App`) for every DB lookup and save, or you will hit `SQLITE_BUSY` / deadlock the writer. The PocketBase docs explicitly call this out. * The cleanest pattern for hooks that need to extend a transaction across `e.Next()` is to temporarily re-point `e.App` at the `txApp` for the duration of the callback so that the downstream chain (which actually saves the withdrawal record) also runs inside the same transaction. Restore it before returning if you want, but the chain semantics expect the propagated app to be the transactional one. * Use `e.Record.GetString("wallet")`, `e.Record.GetFloat("amount")`, and `e.Record.GetString("note")` to read the incoming submission, and `txApp.FindRecordById("wallets", walletId)` to load the wallet inside the transaction. * Use `e.BadRequestError(message, nil)` to short-circuit with a 400 response. Returning a non-nil error from `RunInTransaction` rolls back automatically. * All API rules on both `wallets` and `withdrawals` are left open (`""`) by the bundled migration so that the verifier can interact with them without auth. You do not need to change them. * Build with `go build -o wallet-app .` from the project directory. ## Acceptance Criteria * Project path: /home/user/myproject * Start command: cd /home/user/myproject && ./wallet-app serve --http=0.0.0.0:8090 * Port: 8090 * The PocketBase server must be reachable on port 8090 after start (`GET /api/health` returns HTTP 200). * The `wallets` and `withdrawals` collections (provisioned by the bundled migration) must remain available with the schemas declared by the migration. Do not alter their schemas or API rules. * API endpoints exercised by verification (provided by PocketBase, gated by your hook): * `POST /api/collections/wallets/records`: Used by the verifier to seed wallets. Accepts a JSON body `{"owner": string, "balance": number}` and returns the created wallet record. This endpoint is NOT gated by your hook. * `GET /api/collections/wallets/records/{id}`: Used by the verifier to read the current wallet balance. * `POST /api/collections/withdrawals/records`: Subject to your hook. * Request body shape: ```json { "wallet": string, "amount": number, "note": string } ``` * On success: HTTP `200` with the created withdrawal record (including server-assigned `id`); the referenced wallet's `balance` must have been decreased by exactly the submitted `amount`; the new withdrawal row must be retrievable via `GET /api/collections/withdrawals/records/{id}`. * On any validation failure (non-positive amount, unknown wallet, insufficient balance): HTTP `400`; the wallet's `balance` must be unchanged; no withdrawal row may exist for that request (verified via `GET /api/collections/withdrawals/records?filter=...`). * `GET /api/collections/withdrawals/records`: Standard PocketBase list endpoint used to confirm rejected requests did not persist any row. * Atomicity rule (externally observable): * For every accepted request, the `wallets` table and `withdrawals` table must agree: a successful response always implies both a debited wallet AND a persisted withdrawal row. * For every rejected request, the `wallets` table and `withdrawals` table must both be unchanged.
