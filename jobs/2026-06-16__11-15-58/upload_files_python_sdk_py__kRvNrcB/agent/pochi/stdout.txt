## What the script does

### 1. Configuration (environment variables)
| Variable | Purpose |
|---|---|
| `APIDECK_API_KEY` | Apideck authentication key |
| `APIDECK_APP_ID` | Apideck application ID |
| `APIDECK_CONSUMER_ID` | Apideck consumer ID |
| `APIDECK_SERVICE_ID` | OneDrive connector service ID (e.g. `"onedrive"`) |
| `APIDECK_FILE_STORAGE_DRIVE_NAME` | Display name of the target drive |
| `ZEALT_RUN_ID` | Run identifier embedded in file names and content |

### 2. Drive resolution
Calls `client.file_storage.drives.list(service_id=…)` and iterates pages until it finds the drive whose `.name` matches `APIDECK_FILE_STORAGE_DRIVE_NAME`, capturing its `.id`. Exits with an error if no match is found.

### 3. File upload (upload-session flow)
For each of `report-${run-id}-alpha.txt` and `report-${run-id}-beta.txt`:
1. **Create session** – `upload_sessions.create(name, parent_folder_id="root", size, drive_id, service_id)` → returns an upload session ID.
2. **Upload part** – `upload_sessions.upload(id=session_id, part_number=1, request_body=<bytes>, service_id)` – sends the entire file as one part (both files are tiny).
3. **Finish session** – `upload_sessions.finish(id=session_id, service_id)` → returns a `GetFileResponse` whose `.data.id` is the permanent Apideck file ID.

The SDK transparently routes create/upload/finish calls to `https://upload.apideck.com` (the special upload host) via the per-operation server constants baked into the SDK.

### 4. Log output
Appends a single compact JSON line to `/home/user/myproject/output.log` in the required shape:
```json
{"alpha":{"name":"report-<run-id>-alpha.txt","id":"<apideck-file-id>"},"beta":{"name":"report-<run-id>-beta.txt","id":"<apideck-file-id>"}}
```
