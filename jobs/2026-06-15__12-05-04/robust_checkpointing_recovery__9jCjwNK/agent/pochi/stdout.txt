### How it works

```
bash run.sh <input_file> <output_file> <recovery_dir>
```

**Pipeline stages:**

1. **`StableFileSource`** — reads the CSV line-by-line. Uses a fixed partition key (`"input"`) instead of the file path, so recovery never causes a key-mismatch when the input file changes between runs.
2. **`parse_csv`** — `filter_map` that parses `key,value` lines into `(str, int)` keyed tuples; malformed/empty lines are dropped.
3. **`running_max`** — `stateful_map` with `step_id="running_max"`. State = `max_so_far` per key. Returns `(new_max, new_max)`. This is the state Bytewax persists into SQLite.
4. **`format_output`** — formats each `(key, max)` into a `(key, "key,max")` tuple ready for the sink.
5. **`StableFileSink`** — writes each string value to the output file. Also uses a fixed partition key (`"output"`), truncating the file fresh on every run.

**Recovery mechanics:**

- On first run: `python -m bytewax.recovery <recovery_dir> 1` creates `part-0.sqlite3`.
- On subsequent runs: `python -m bytewax.run` with `-r <recovery_dir> -s 10 -b 0` resumes from the SQLite snapshot — the `stateful_map` state (running maximums per key) is restored automatically.
