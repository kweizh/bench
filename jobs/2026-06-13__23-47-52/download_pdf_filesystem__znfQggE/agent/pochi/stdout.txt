# Download and Persist a PDF with the Capacitor Filesystem Plugin ## Background You are building a small Vite + TypeScript web application that will eventually be packaged as a native mobile app via Capacitor v8. The Capacitor Filesystem plugin gives apps a NodeJS-like API for working with the device's storage. On the web target, this plugin transparently falls back to a persistent IndexedDB-backed store so that the same JavaScript code that runs on iOS and Android also works in the browser-based preview that this task uses for verification. A minimal Vite + TypeScript project has already been scaffolded for you at `/home/user/myapp`. The project also contains a small fixture document at `public/sample.pdf` that Vite will copy into `dist/` during the build. Your job is to integrate Capacitor v8, install `@capacitor/filesystem`, and implement a small UI that downloads this PDF over HTTP, persists it to the Documents directory through Capacitor's Filesystem API, then reads it back and surfaces the file's size and SHA-256 fingerprint to the page so the user can verify the saved copy. ## Requirements * Integrate Capacitor v8 into the existing Vite project using the non-interactive CLI flow. The Capacitor app must be configured with: * App name: `Filesystem Demo` * Application/package id: `com.example.fsdemo` * Web assets directory aligned with Vite's build output (`dist`). * Install and use the `@capacitor/filesystem` plugin (version compatible with Capacitor v8) for persistence. You may NOT shortcut the requirement by writing the bytes directly to `localStorage`, `IndexedDB`, or `window.fetch`-backed memory caches — the persisted file MUST go through `Filesystem.writeFile` with the `Directory.Documents` directory. * Implement a UI in `index.html` (and any TypeScript modules under `src/`) that includes: * A button with the HTML id `download-pdf` that, when clicked, downloads the file served at `/sample.pdf` (relative to the same origin) and saves it through Capacitor Filesystem at the path `sample.pdf` under `Directory.Documents`. * A status region with the HTML id `download-status` whose text content updates to one of the following strings, in order: * Before any download has been attempted: `idle`. * While a download is in progress: `downloading`. * On success: `saved`. * On error: `error: <message>` (where `<message>` is a short, human-readable description). * An element with the HTML id `file-size` that, on success, displays the saved file's size in bytes as a plain decimal integer (no commas, no units). * An element with the HTML id `file-sha256` that, on success, displays the lowercase hex SHA-256 digest of the saved file (64 hex characters). * The displayed size and SHA-256 must be computed by reading the file BACK from Capacitor Filesystem (i.e., via `Filesystem.readFile` / `Filesystem.stat`), not from the in-memory bytes that were just downloaded. This proves the file truly persisted. * `npx cap sync` must run successfully against the produced web build (no native platforms need to be added). ## Implementation Hints * Use `npx cap init` with positional `appName` and `appId` arguments and the `--web-dir` flag to avoid the interactive prompt. * Vite's default build output directory is `dist`; ensure it matches `webDir` in `capacitor.config.ts` (or `.json`). * The Capacitor Filesystem plugin's web implementation persists files in IndexedDB, so no native runtime is needed for the verification environment. * For binary data, the Filesystem API expects base64-encoded strings when no `encoding` is provided. You can convert an `ArrayBuffer` to base64 with the `FileReader` API (`readAsDataURL` and slice off the data-URL prefix) or with `btoa(String.fromCharCode(...new Uint8Array(...)))` for small files. * The Web Crypto API (`crypto.subtle.digest('SHA-256', buffer)`) is available in the preview environment and is the easiest way to compute a SHA-256 hex digest. * Make sure the script that wires up the button is loaded as an ES module so that the dynamic import of `@capacitor/filesystem` succeeds. ##
