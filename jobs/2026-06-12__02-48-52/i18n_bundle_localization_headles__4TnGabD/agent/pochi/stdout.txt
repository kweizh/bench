Bundles are loaded from a `FileHandle` (typically `Gdx.files.internal("i18n/Messages")`), which makes the feature easy to drive under the non-rendering **headless** backend (`com.badlogicgames.gdx:gdx-backend-headless`) because no GPU, audio, or input plumbing is involved. ## Key design decisions

| Concern | Solution |
|---|---|
| HeadlessApplication threading | `CountDownLatch(1)` in `create()` â€“ `Launcher.main` blocks on `awaitCompletion()` until the command loop finishes, then calls `headlessApp.exit()` |
| No GL context | `HeadlessApplicationConfiguration.updatesPerSecond = -1` disables the render loop; `Gdx.gl*` is never touched |
| Bundle loading | `I18NBundle.createBundle(Gdx.files.internal("i18n/Messages"), new Locale(code))` for each of `en`/`fr`/`de` |
| Base-bundle fallback | libGDX's `I18NBundle` inherits Java's `ResourceBundle` fallback chain, so a French bundle missing `gameName` reads it from `Messages.properties` automatically |
| Missing key detection | `safeGet`/`safeFormat` catch `java.util.MissingResourceException` and return `null` |
| assets on classpath | `sourceSets.main.resources.srcDirs += ["assets"]` bakes the `i18n/` tree into the JAR |
