# Cancel a Triggered Knock Workflow Before Email Delivery ## Background Knock (https://knock.app) is a notification infrastructure platform whose workflow engine orchestrates cross-channel messaging from a single trigger. Knock supports **workflow cancellation**: if a triggered workflow contains a step that pauses execution (such as a `delay`), the workflow run can be aborted before the downstream channel steps deliver any messages. To enable this, the trigger request must include a `cancellation_key`, and a subsequent `POST /v1/workflows/{key}/cancel` request must reference the same `cancellation_key`. The Knock workspace used for this task already has two channels configured: - An in-app feed channel with channel key `in-app`. - A Mailtrap email channel with channel key `mailtrap`. You will build a Node.js automation that creates a delayed email workflow, triggers it for a Gmail-backed recipient with a `cancellation_key`, and then cancels the triggered run before the delay window elapses so that the email is **never delivered**. ## Requirements - Implement the automation as a Node.js project located at `/home/user/cancel_task` that runs end-to-end with `npm start`. - Use the `@knocklabs/mgmt` SDK (authenticated with `KNOCK_SERVICE_TOKEN`) to **upsert** a workflow in the `development` environment whose body contains: 1. A `delay` function step that pauses the workflow long enough for cancellation to be issued (use a delay of at least 30 seconds). 2. An email `channel` step that sends a Mailtrap email after the delay using channel key `mailtrap`. The sender address must use the Mailtrap domain restricted by `MAILTRAP_DOMAIN`. The subject and HTML body templates must render the recipient's name. - **Activate** the workflow in the `development` environment via the Management API after upserting. - Use the `@knocklabs/node` SDK (authenticated with `KNOCK_API_TOKEN`) to **trigger** the workflow exactly once with: - A single inline recipient whose `email` is a Gmail `+receiver-<run-id>` address (so we can confirm the email did NOT arrive). - A `cancellation_key` that uniquely identifies this run. - After triggering, wait long enough that the cancellation request will not overtake the trigger (Knock recommends at least 5 seconds between trigger and cancel), then call the Knock cancel workflow API with the same workflow key and `cancellation_key` to abort the run before the delay completes. - Persist the workflow key, workflow run id (returned by the trigger call), cancellation key, and recipient email to a log file in the format described in the Acceptance Criteria. - All resource identifiers that are externally visible must be suffixed with the current `run-id` so concurrent runs do not collide. ## Implementation Hints - Read the current `run-id` from the `ZEALT_RUN_ID` environment variable and append it to the workflow key, recipient id, cancellation key, and email subject so concurrent runs do not collide. - The Knock Management API only writes to the `development` environment; pass `environment: "development"` to both `workflows.upsert` and the activation call. - The workflow must include both a `delay` function step **and** an email `channel` step in that order. The `delay.settings.delay_for` shape uses `{ unit, value }` (for example `{ unit: "seconds", value: 30 }`). Cancelling only works because the delay pauses the run before the email is dispatched. - The email step's sender address must be the Mailtrap-allowed address `sender-<run-id>@<MAILTRAP_DOMAIN>` configured via the email step's `settings.from_email_address`. - The Knock `cancellation_key` must be supplied on the trigger call (top-level field) and again on the cancel call. Use a key derived from the `run-id` so the cancel API can match the queued run. - The Knock cancel API returns `204 No Content` on success; do not treat the empty body as an error. - The script SHOULD wait at least 5 seconds between trigger and cancel and exit before the workflow's delay elapses — otherwise the email will be sent. ## Acceptance Criteria - Project path: `/home/user/cancel_task` - Ensure the real Knock workflow upsert, activation, trigger, and cancel API calls are executed against the live Knock environment (no mocks). - Log file: `/home/user/cancel_task/output.log` - Read `run-id` from the `ZEALT_RUN_ID` environment variable. - A workflow whose `key` equals `cancel-flow-<run-id>` MUST exist in the `development` environment of the Knock workspace and MUST be `active`. - The workflow MUST contain at least one `delay` function step followed by a `channel` step whose `channel_key` is `mailtrap`. - The workflow MUST be triggered exactly once with a single inline recipient whose: - `id` is `user-<run-id>`. - `email` is `<GMAIL_USER_NAME>+receiver-<run-id>@gmail.com`. - `name` is `User <run-id>`. - The trigger call MUST include a `cancellation_key` equal to `cancel-<run-id>`. - The cancel API MUST be called with the same workflow key and `cancellation_key` after the trigger so that the email is never delivered. - The log file MUST contain the following lines (one per line, in any order): - `Workflow Key: cancel-flow-<run-id>` - `Workflow Run ID: <uuid>` where `<uuid>` is the `workflow_run_id` returned by the trigger call. - `Cancellation Key: cancel-<run-id>` - `Recipient Email: <GMAIL_USER_NAME>+receiver-<run-id>@gmail.com` - No email matching the workflow's subject for this recipient may arrive in the Gmail inbox of `<GMAIL_USER_NAME>+receiver-<run-id>@gmail.com`.
