# libGDX Headless: Scripted Keystroke Replay Walker ## Background libGDX ships a non-rendering backend (`com.badlogicgames.gdx:gdx-backend-headless`) that swaps the live `Graphics`/`Input`/`Audio` implementations for mocks so that a libGDX `ApplicationListener` can be driven from a server, CI runner, or test harness. The bundled `MockInput` simply returns zero/`false` for every method, which makes it useless on its own for any input-driven game. A common pattern is to subclass `MockInput`, read a sequence of keystrokes from a text file, and re-dispatch them to the live `InputProcessor` each tick so that a deterministic replay of player input can drive the game from a script. In this task you will build a small libGDX 1.14.2 project around the **headless** backend that simulates a 2D grid walker. ## Key Design Decisions

1. **ScriptedInput** extends `MockInput` and overrides `setInputProcessor()`/`getInputProcessor()` because `MockInput.setInputProcessor()` is a no-op (empty method body) and `getInputProcessor()` always returns a default `InputAdapter`. Without this override, the input processor chain is broken.

2. **InputProcessor registration** is deferred to the first `render()` call (not `create()`) to avoid a race condition: `HeadlessApplication`'s constructor starts the main loop thread immediately, so `Gdx.input` must be overwritten with `ScriptedInput` after construction but before the first `setInputProcessor()` call.

3. **Launcher blocking**: The main thread polls `scriptedInput.isExhausted()` with `Thread.sleep(100)` and waits an extra 300ms grace period after exhaustion to let `Gdx.app.exit()` propagate.
