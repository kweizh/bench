This is the foundation for collaborative agents, group standups, team support bots, and shared planning assistants. The CLI must use the official `@alchemystai/sdk` package and target the real Alchemyst AI API — no mocks, no stubs, no hardcoded responses. ## How it works
- **Authentication**: reads `ALCHEMYST_AI_API_KEY` from the environment and passes it to `new AlchemystAI({ apiKey })`.
- **Shared `sessionId`**: derived as `team-standup-${ZEALT_RUN_ID}` so parallel evaluation runs cannot collide; all invocations inside a run land in the same shared thread.
- **Args parsing**: manual `process.argv` walk supporting `--user-id`, `--add`, `--query` (with both space- and `=`-separated forms).
- **Fast-fail validation**: if `--user-id` is missing the CLI exits non-zero with a message containing the literal string `MISSING_PARAMETERS`, mirroring the Alchemyst API error code (both `userId` and `sessionId` are mandatory for memory operations).
- **`--add`**: calls `client.v1.context.memory.add({ sessionId, contents: [{ content, role: "user", metadata: { messageId, userId }, userId }], metadata: { groupName: [sessionId], userId, sessionId } })`. The `userId` is attached as per-message metadata and the shared `sessionId` is used as the groupName so any teammate's writes land in the same retrievable namespace. Prints `ADDED: <content>` on success.
- **`--query`**: calls `client.v1.context.search({ query, similarity_threshold: 1.0, minimum_similarity_threshold: 0.0, scope: "internal", body_metadata: { groupName: [sessionId], sessionId }, user_id })`. Thresholds are wide-open so semantically loose queries reliably hit the stored entries; the `groupName: [sessionId]` filter scopes the search to the shared session thread, so memories written by any `userId` in the run are surfaced. Each returned entry is printed on its own line as `MEMORY: <content>` (it iterates over `memories`, `contexts`, and `results` arrays for forward-compat with the response shape).
